AIMG - проект для продаже пользователям доступа к API для генерации видео и картинок с разными целями

# **1\. Цель и общий обзор**
Хотим сделать backend-обёртку для интеграции AI сервисов генерации картинок (и возможно видео в будущем).
Т.е. сценарий такой. Берем API, которое позволяет генерировать / редактировать картинки или видео. Пишем для этого API какую-то job обработки внутри нашего backend. Даём партнёру API key, и партнёр встраивает к себе эту job, и гонит к нам трафик. Мы на своей стороне управляем запусками этих джобов: проверяем лимиты, тарифы, храним картинки и промежуточные артефакты, управляем приоритетами API.

Короче нам по сути нужна базовая механика с апишкой, интеграциями, партнерами, картинками, джобами, админкой, и возможность добавлять разные реализации этих джобов.
Сервис предназначен для встраивания партнерами в собственные фронты (Telegram-боты, сайты и т.п.). Сервис ведет историю запросов/ответов, учитывает бесплатные и оплаченные лимиты и возвращает результат обработки в виде файла.

# **2\. Область работ и границы проекта**

## **2.1 Входит в объем работ**

·       REST API сервиса (версия v1) для: загрузки изображения, получения результата/статуса, получения баланса/лимитов, получения истории.  
·       Авторизация партнеров через API-ключи.  
·       Модель данных: партнеры, интеграции, пользователи, задания (jobs), биллинг/кредиты, аудит.  
·       Хранение истории: оригинал изображения (или видео) \+ параметры запроса \+ результат обработки \+ статус/ошибки.  
·       Админ-панель (внутренняя) для команды владельца сервиса: управление партнерами/интеграциями/ключами, просмотр и поиск jobs, управление кредитами пользователей, экспорт истории.  
·       Механизм списания запросов: N бесплатных запросов на пользователя, далее — списание из купленных кредитов; при неуспешной обработке списание не производится.  
·       Возможность организовать Fallback по моделям внутри jobs: при ошибке первого провайдера — пробовать следующий (конфигурируемый список).  
·       Локализация системных текстов (RU/EN) с возможностью добавления языков без изменения бизнес-логики.

## **2.2 Не входит в объем работ (пока)**

·       Реализация платежного шлюза/приема платежей (оплата на стороне партнера или отдельного компонента).  
·       Фронтенды партнеров (Telegram-боты, сайты и др.).  
·       Контент-фильтры/NSFW-модерация.  
·       Пакетная обработка (несколько изображений одним запросом), альбомы/очереди от одного пользователя (может быть добавлено позднее).  
·       Политики удаления/ретеншн результатов (хранение бессрочно по умолчанию).

# **4\. Пользовательские сценарии (user stories)**

4.1 Создание задания на обработку

1\. 	Партнер вызывает API создания job и передает изображение \+ текстовый prompt (опционально) \+ язык пользователя (в зависимости от )
2\. 	Сервис валидирует формат/размер, проверяет лимиты (free/paid).  
3\. 	Если кредитов недостаточно: возвращает status=needs\_payment (без списания) и причины.  
4\. 	Если кредитов достаточно: создает job, запускает обработку через провайдера(ов), по готовности сохраняет результат и возвращает ссылку/файл.

4.2 Получение результата

5\. 	Партнер опрашивает статус job (polling) до получения succeeded/failed/needs\_payment.  
6\. 	При succeeded партнер скачивает результат и отдает пользователю в своем фронте.  
7\. 	При failed партнер показывает локализованную ошибку, списание не производится.

# **5\. Архитектура (логическая)**

Сервис состоит из API слоя, слоя бизнес-логики (лимиты/биллинг/история), очереди задач (рекомендовано), воркеров обработки и адаптеров к провайдерам нейросетей. Файлы (оригиналы и результаты) хранятся во внешнем объектном хранилище.

## **5.1 Компоненты**

·       API Gateway / REST API (v1) — прием запросов от партнеров, авторизация, валидация.  
·       DB — хранение партнеров, интеграций, пользователей, jobs, балансов, аудита.  
·       Object Storage — хранение файлов (оригиналы и результаты). Рекомендуется S3-compatible интерфейс.  
·       Queue (опционально, но рекомендовано) — очередь задач для воркеров.  
·       Workers — выполнение обработки через провайдеров нейросетей, fallback.  
·       Admin Panel — внутренняя админка для команды владельца.

# **6\. Нефункциональные требования**

| Параметр      | Требование                                                                                                                |
| :------------ | :------------------------------------------------------------------------------------------------------------------------ |
| Надежность    | Повторные запросы клиента/партнера не должны приводить к двойному списанию кредитов.                                      |
| Безопасность  | Авторизация партнеров по API-ключам; разграничение доступа к данным партнера; управление доступами на уровне разных jobs. |
| Наблюдаемость | Логи, метрики, трассировка по request\_id/job\_id.                                                                        |

# **7\. Хранение данных и файлов**

## **7.1 База данных (DB)**

Тип СУБД не фиксируется (PostgreSQL рекомендуется). Требование: транзакционность и поддержка индексов для поиска по job\_id, user\_id, partner\_id, датам и статусам.

## **7.2 Внешнее файловое хранилище**

Файлы (оригиналы и результаты) хранятся во внешнем объектном хранилище. Рекомендуемый интерфейс: S3-compatible (AWS S3/MinIO/аналоги). Сервис хранит в DB только ссылки/ключи объектов и метаданные.
·       Ретеншн: бессрочно (по умолчанию), возможность внедрения удаления — отдельной задачей.

# **8\. Ограничения на входные данные**

Ограничения по форматам и размерам должны быть конфигурируемыми (без перекомпиляции). На старте допускается использовать дефолтные значения и возвращать ошибку "изображение не подходит" при нарушении ограничений.

| Параметр               | Дефолт (можно изменить)                                                                   | Комментарий                                       |
| :--------------------- | :---------------------------------------------------------------------------------------- | :------------------------------------------------ |
| Макс. размер файла     | задается для каждой job                                                                   | Конфиг, можно поднять позже.                      |
| Поддерживаемые форматы | в зависимости от каждой job это может быть набор картиночных или видео форматов или текст | HEIC/доп. форматы — опционально.                  |

# **9\. Локализация (i18n)**

Если будет необходимость возвращать в API тексты, необходимо поддержать локализацию на разные языки. На старте: RU, EN. Пользователь выбирает язык на стороне партнера и партнер передает код языка в API. Тексты ответов должны храниться в ресурсах/таблицах/файлах локализации и быть расширяемыми для добавления новых языков.

# **10\. Биллинг и лимиты**

## **10.1 Правила списания**

·       Каждый пользователь имеет некоторое количество бесплатных запроса (free credits) на старте.  
·       Далее используются оплаченные кредиты (paid credits).  
·       Списание выполняется только при результате succeeded.  
·       При failed/needs\_payment кредиты не списываются.  
·       Баланс не может становиться отрицательным.

## **10.2 Пополнение/начисления**

Оплата реализуется вне сервиса (платежный шлюз/партнер). Сервис должен поддерживать начисление кредитов пользователю: а) через админку вручную (акции/компенсации), б) через API endpoint для интеграции с платежным шлюзом или партнёром (опционально).

# **11\. Авторизация и безопасность**

## **11.1 Авторизация партнеров**

Партнеры авторизуются по API Key, передаваемому в заголовке X-API-Key. Ключи управляются через админку. Эти API Key формируются на стороне нашего бекенда, в формате JWT.

## **11.2 Разграничение доступа**

Данные пользователей и jobs изолированы по partner\_id. Один партнер не может получить доступ к данным другого.

## **11.3 Идемпотентность и защита от задвоений**

·       Endpoint создания job принимает заголовок Idempotency-Key (обязательный).  
·       При повторном запросе с тем же ключом сервис возвращает ранее созданный job и не выполняет повторное списание.  

## **11.4 Персональные данные**

Сервис не хранит персональные данные кроме идентификаторов (telegram\_user\_id или external\_user\_id).

# **12\. API спецификация (v1) — очень примерно**

API реализуется как REST. Должен быть опубликован OpenAPI/Swagger документ. Ниже приведен целевой набор эндпоинтов. Конкретные поля могут быть уточнены на этапе проектирования.

## **12.1 Базовые модели ответа**
·       Все ответы должны содержать поля: request\_id, success, error (если есть).

## **12.2 Эндпоинты (черновой перечень)**

| Метод | Путь                      | Назначение                        | Примечания                                                                                                                                                                |
| :---- | :------------------------ | :-------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| GET   | /health                   | Проверка работоспособности        | Без авторизации (опционально).                                                                                                                                            |
| GET   | /v1/meta/languages        | Список поддерживаемых языков      | RU/EN \+ расширяемо.                                                                                                                                                      |
| POST  | /v1/jobs                  | Создать задание на обработку      | multipart upload или file\_url; Idempotency-Key обязателен. Возможно необходимо выделить загрузку файла в отдельный метод, чтобы отделить эту логику на отдельный сервер. |
| GET   | /v1/jobs/{job\_id}        | Получить статус/результат задания | Polling партнера.                                                                                                                                                         |
| GET   | /v1/jobs/{job\_id}/result | Скачать результат                 | Подписанная ссылка на cdn/s3 или прокси-скачивание.                                                                                                                       |
| GET   | /v1/users/me/balance      | Баланс/лимиты пользователя        | По user идентификатору из запроса партнера.                                                                                                                               |
| GET   | /v1/users/me/history      | История обработок пользователя    | Пагинация.                                                                                                                                                                |
| POST  | /v1/billing/topup         | Начислить кредиты пользователю    | Опционально (интеграция с платежами).                                                                                                                                     |

## **12.4 Статусы job**

| Статус         | Описание                                                     |
| :------------- | :----------------------------------------------------------- |
| queued         | Принято, ожидает обработки.                                  |
| running        | В обработке.                                                 |
| succeeded      | Успешно обработано, результат доступен.                      |
| failed         | Ошибка обработки (кредиты не списаны).                       |
| needs\_payment | Недостаточно кредитов/нужно пополнение (кредиты не списаны). |

## **12.5 Ошибки**

·       Ошибки должны возвращаться с кодом и локализуемым сообщением (ru/en)

| Код | HTTP | Когда возникает |
| :---- | :---- | :---- |
| INVALID\_IMAGE | 400 | Неверный формат/размер/битый файл; "изображение не подходит". |
| UNAUTHORIZED | 401 | Нет или неверный API key. |
| FORBIDDEN | 403 | Доступ к ресурсу другого партнера. |
| NOT\_FOUND | 404 | Job/User не найден. |
| NEEDS\_PAYMENT | 402 или 200\* | Нет кредитов. (\*Можно 200 с status=needs\_payment по договоренности.) |
| PROVIDER\_ERROR | 502 | Ошибка провайдера нейросети (после попыток fallback). |
| TIMEOUT | 504 | Обработка превысила лимит времени. |
| INTERNAL | 500 | Неожиданная ошибка сервиса. |

# **13\. Админ-панель (внутренняя)**

## **13.1 Роли**

·       Admin: полный доступ.  
·       Viewer(User): просмотр событий/метрик/истории, без изменений.

## **13.2 Разделы админки (минимум)**

·       Партнеры: список, создание/редактирование, статус (active/blocked).  
·       Интеграции партнера: список, создание, выдача API ключей, ротация/отзыв.  
·       Пользователи: поиск по telegram\_user\_id/external\_user\_id, просмотр баланса (free/paid), история jobs.  
·       Jobs: таблица с фильтрами (дата, статус, партнер, интеграция, user\_id), просмотр деталей (включая ссылки на файлы).  
·       Кредиты: ручное начисление/корректировка (с обязательным комментарием причины).  
·       Экспорт: выгрузка истории (CSV) с фильтрами; опционально JSON.

# **14\. Логи, аудит и экспорт**
Сервис должен вести аудит административных действий (кто/когда/что изменил), а также хранить технические логи запросов.
·       Экспорт истории: CSV (обязательно), JSON (желательно).

# **15\. Развертывание и конфигурация**
·       Docker-first: сервис поставляется в виде Docker image \+ docker-compose
·       Конфигурация через переменные окружения: DB, object storage, ключи провайдеров, лимиты файлов, список провайдеров для fallback, локализации.  
·       OpenAPI спецификация должна быть доступна по /docs или отдельным файлом.

# **16\. Критерии приемки**

8\. 	Партнер может создать job, получить succeeded и скачать результат.  
9\. 	При отсутствии кредитов сервис возвращает needs\_payment и не создает списаний.  
10\.  При failed кредиты не списываются.  
11\.  Idempotency-Key предотвращает дублирование job и двойные списания.  
12\.  Админка позволяет: управлять партнерами/ключами, просматривать jobs, начислять кредиты пользователям и экспортировать историю.  
13\.  RU/EN сообщения доступны и выбираются через параметр language.
